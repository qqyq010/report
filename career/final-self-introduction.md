# Final Self Introduction

## 1. 제출용 자소서

상태 변경, 실패 복구, 검증 경계. 세 프로젝트를 정리하며 Unity/C# 클라이언트 개발에서 가장 중요하게 본 기준입니다. 기능이 화면에서 정상처럼 보여도 실제 데이터가 같은 시점에 바뀌지 않거나, 실패했을 때 복구 흐름이 없으면 이후에 설명하기 어려운 문제가 됩니다. 팀 프로젝트를 진행하면서 코드만 확인한 것과 실행까지 확인한 것을 섞어 말하면 협업에서도 혼선이 생긴다는 점을 느꼈습니다. 그래서 결과를 먼저 내세우기보다 어떤 상태가 언제 바뀌는지, 실패했을 때 어디로 돌아가는지, 무엇을 확인했고 무엇이 남아 있는지를 나누어 보려고 했습니다.

팀프로젝트1 Coin Laundry에서는 Inventory와 Shop 흐름을 보며 이 문제를 가장 직접적으로 다뤘습니다. 인벤토리에서 아이템을 잡고 움직이는 preview 상태와 실제 데이터에 반영되는 commit 상태가 같은 시점에 처리되면, 화면에 보이는 위치와 실제 인벤토리 상태가 어긋날 수 있다고 보았습니다. 상점 구매도 단순히 골드를 차감하고 아이템을 지급하는 성공 흐름만 보면 부족했습니다. 아이템 수령이 실패했을 때 골드만 빠지는 상황을 막으려면 차감, 수령, 환불 흐름을 함께 봐야 했습니다. merge도 UI가 모든 판단을 맡으면 판정과 실행 책임이 섞일 수 있어 서비스 기준으로 나누어 설명할 수 있게 정리했습니다. 다만 팀 프로젝트 자료이므로 인벤토리/상점 전체를 단독 구현한 것처럼 쓰지 않고, 담당 범위 후보와 문서로 설명 가능한 구조까지만 표현했습니다.

Project3 Fisher에서는 비슷한 데이터와 상태 문제를 더 큰 협업 구조 안에서 보았습니다. 이 프로젝트는 완성작으로 내세우기보다, 무엇이 확인된 자료이고 무엇이 아직 확인이 필요한지 나누는 과정에 집중했습니다. Fisher/CSH, PlayFab, CloudScript, 밸런스, UI 리소스가 함께 연결되면 한쪽 코드만 보고 원인을 확정하기 어렵습니다. 특히 서버와 연결된 문제는 Unity에서 어떤 요청을 보냈는지, 서버가 어떤 응답을 돌려줬는지, 실제 환경에 반영된 근거가 있는지를 따로 봐야 했습니다. Codex/AI도 성과 표현을 키우는 도구가 아니라, 읽기 전용 확인, 영향 범위 점검, 문서 정리와 확인 목록 관리에 제한해 사용했습니다. 또 공개해도 되는 요약과 Drive에 남겨야 할 원본 자료를 나누며 팀 자료를 다룰 때의 경계를 의식했습니다.

개인 프로젝트 EscapeRoom은 팀 프로젝트에서 조심스럽게 말해야 하는 직접 구현 부분을 Unity/C# 코드로 보강한 사례입니다. GameManager와 ProgressState를 나누어 현재 행동 가능 상태와 퍼즐 진행도를 분리했고, EnemyVision에서는 거리, 각도, Raycast 순서로 감지 조건을 나누었습니다. 감지 결과는 DetectionController에서 FailureHandler로 이어지고, 실패 뒤 복귀는 CheckpointManager와 CheckpointTrigger 흐름으로 정리했습니다. Treasure Puzzle에서는 grid, path, reset, phase unlock과 퍼즐 내부 추적 적의 실패 조건을 나누어 관리했습니다. 이 부분은 상태 관리, 감지, 실패 처리, 복귀, 퍼즐 진행을 각각 다른 코드 경로로 설명할 수 있습니다. 다만 scene 연결, Play Mode, 영상 자료는 아직 더 확인이 필요하므로 코드 구조를 설명할 수 있는 범위로만 사용합니다.

입사 후에는 기능을 빠르게 붙이는 데서 멈추지 않고, 상태 변경과 실패 복구가 설명 가능한 클라이언트 코드를 작성하고 싶습니다. 회사가 콘텐츠를 확장하고 팀 단위로 서비스를 운영할수록 UI, 데이터, 서버 응답, 확인 기록이 어긋날 가능성도 커진다고 생각합니다. 그런 환경에서 재현 가능한 작업 단위와 공유 가능한 문서화를 남기며, 협업 속에서도 안정적으로 확장되는 Unity/C# 클라이언트 개발에 기여하겠습니다.

## 2. 검토 메모

### 글자 수

* 공백 포함: 1925
* 공백 제외: 1508

### 사용한 소재

* Coin Laundry: Inventory/Shop 흐름, preview/commit 분리, spend/receive/refund 흐름, merge 판정/실행 책임 분리.
* Project3 Fisher: 확인한 자료와 확인 필요 자료 구분, Unity 요청/서버 응답/실제 반영 근거, 공개 가능 요약과 Drive 원본 분리, Codex/AI 사용 경계.
* EscapeRoom: GameState/ProgressState, EnemyVision 거리/각도/Raycast, FailureHandler/Checkpoint, Treasure Puzzle.
* 입사 후 방향: 상태 변경과 실패 복구가 설명 가능한 클라이언트 코드, 재현 가능한 작업 단위, 협업 가능한 문서화.

### 제한 표현으로 낮춘 부분

* Coin Laundry는 직접 구현 확정이 아니라 담당 범위 후보와 설명 가능한 구조로 표현했습니다.
* Project3 Fisher는 완성작이 아니라 확인한 자료와 확인 필요 자료를 나누는 흐름으로 표현했습니다.
* EscapeRoom은 코드 구조 설명으로 제한하고 scene, Play Mode, 영상 검증 완료 표현은 쓰지 않았습니다.

### 제출 전 확인 필요

* Coin Laundry: 작성자/커밋, 팀 승인, Play Mode 원본 로그, EventBus 또는 이벤트 연결 코드, 시연 영상.
* Project3 Fisher: 실제 담당 범위, `.cs` 근거, CloudScript handler, Unity caller, 라이브 반영 근거, balance workbook, UI 적용 근거.
* EscapeRoom: scene/prefab/Inspector 연결, Play Mode 또는 빌드 검증, 시연 영상, HideZone/PuzzleSafeZone/Player 이동 코드.
